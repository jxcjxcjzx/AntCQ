      <span class="octicon octicon-book"></span>
      README.md
    </h3>

    <article class="markdown-body entry-content" itemprop="mainContentOfPage"><h1><a id="user-content-hlint--" class="anchor" href="#hlint--" aria-hidden="true"><span class="octicon octicon-link"></span></a>HLint <a href="https://hackage.haskell.org/package/hlint"><img src="https://camo.githubusercontent.com/87b9974f6de9bb04c012da60a7069cced7d955b2/68747470733a2f2f696d672e736869656c64732e696f2f6861636b6167652f762f686c696e742e7376673f7374796c653d666c6174" alt="Hackage version" data-canonical-src="https://img.shields.io/hackage/v/hlint.svg?style=flat" style="max-width:100%;"></a> <a href="https://travis-ci.org/ndmitchell/hlint"><img src="https://camo.githubusercontent.com/ff3004de0f401ae1116af801756a77c8febe7a9a/68747470733a2f2f696d672e736869656c64732e696f2f7472617669732f6e646d69746368656c6c2f686c696e742e7376673f7374796c653d666c6174" alt="Build Status" data-canonical-src="https://img.shields.io/travis/ndmitchell/hlint.svg?style=flat" style="max-width:100%;"></a></h1>

<p>HLint is a tool for suggesting possible improvements to Haskell code. These suggestions include ideas such as using alternative functions, simplifying code and spotting redundancies. You can try HLint online at <a href="http://lpaste.net/">lpaste.net</a> - suggestions are shown at the bottom. This document is structured as follows:</p>

<ul>
<li><a href="#installing-and-running-hlint">Installing and running HLint</a></li>
<li><a href="#faq">FAQ</a></li>
<li><a href="#customizing-the-hints">Customizing the hints</a></li>
</ul>

<h3><a id="user-content-acknowledgements" class="anchor" href="#acknowledgements" aria-hidden="true"><span class="octicon octicon-link"></span></a>Acknowledgements</h3>

<p>This program has only been made possible by the presence of the <a href="https://github.com/haskell-suite/haskell-src-exts">haskell-src-exts</a> package, and many improvements have been made by <a href="http://www.nbroberg.se">Niklas Broberg</a> in response to feature requests. Additionally, many people have provided help and patches, including Lennart Augustsson, Malcolm Wallace, Henk-Jan van Tuyl, Gwern Branwen, Alex Ott, Andy Stewart, Roman Leshchinskiy, Johannes Lippmann, Iustin Pop, Steve Purcell and others.</p>

<h3><a id="user-content-bugs-and-limitations" class="anchor" href="#bugs-and-limitations" aria-hidden="true"><span class="octicon octicon-link"></span></a>Bugs and limitations</h3>

<p>Bugs can be reported <a href="https://github.com/ndmitchell/hlint/issues">on the bug tracker</a>. There are some issues that I do not intend to fix:</p>

<ul>
<li>HLint operates on each module at a time in isolation, as a result HLint does not know about types or which names are in scope.</li>
<li>The presence of <code>seq</code> may cause some hints (i.e. eta-reduction) to change the semantics of a program.</li>
<li>Either the monomorphism restriction, or rank-2 types, may cause transformed programs to require type signatures to be manually inserted.</li>
<li>The <code>RebindableSyntax</code> extension can cause HLint to suggest incorrect changes.</li>
<li>HLint turns on many language extensions so it can parse more documents, occasionally some break otherwise legal syntax - e.g. <code>{-#INLINE foo#-}</code> doesn't work with <code>MagicHash</code>. These extensions can be disabled with <code>-XNoMagicHash</code>.</li>
</ul>

<h2><a id="user-content-installing-and-running-hlint" class="anchor" href="#installing-and-running-hlint" aria-hidden="true"><span class="octicon octicon-link"></span></a>Installing and running HLint</h2>

<p>Installation follows the standard pattern of any Haskell library or program, type <code>cabal update</code> to update your local hackage database, then <code>cabal install hlint</code> to install HLint.</p>

<p>Once HLint is installed, run hlint source where source is either a Haskell file, or a directory containing Haskell files. A directory will be searched recursively for any files ending with .hs or .lhs. For example, running HLint over darcs would give:</p>

<pre><code>$ hlint darcs-2.1.2

darcs-2.1.2\src\CommandLine.lhs:94:1: Error: Use concatMap
Found:
  concat $ map escapeC s
Why not:
  concatMap escapeC s

darcs-2.1.2\src\CommandLine.lhs:103:1: Warning: Use fewer brackets
Found:
  ftable ++ (map (\ (c, x) -&gt; (toUpper c, urlEncode x)) ftable)
Why not:
  ftable ++ map (\ (c, x) -&gt; (toUpper c, urlEncode x)) ftable

darcs-2.1.2\src\Darcs\Patch\Test.lhs:306:1: Error: Use a more efficient monadic variant
Found:
  mapM (delete_line (fn2fp f) line) old
Why not:
  mapM_ (delete_line (fn2fp f) line) old

... lots more suggestions ...
</code></pre>

<p>Each suggestion says which file/line the suggestion relates to, how serious the issue is, a description of the issue, what it found, and what you might want to replace it with. In the case of the first hint, it has suggested that instead of applying <code>concat</code> and <code>map</code> separately, it would be better to use the combination function <code>concatMap</code>.</p>

<p>The first suggestion is marked as an error, because using <code>concatMap</code> in preference to the two separate functions is always desirable. In contrast, the removal of brackets is probably a good idea, but not always. Reasons that a hint might be a warning include requiring an additional import, something not everyone agrees on, and functions only available in more recent versions of the base library.</p>

<p><strong>Bug reports:</strong> The suggested replacement should be equivalent - please report all incorrect suggestions not mentioned as known limitations.</p>

<h3><a id="user-content-reports" class="anchor" href="#reports" aria-hidden="true"><span class="octicon octicon-link"></span></a>Reports</h3>

<p>HLint can generate a lot of information, making it difficult to search for particular types of errors. The <code>--report</code> flag will cause HLint to generate a report file in HTML, which can be viewed interactively. Reports are recommended when there are more than a handful of hints.</p>

<h3><a id="user-content-language-extensions" class="anchor" href="#language-extensions" aria-hidden="true"><span class="octicon octicon-link"></span></a>Language Extensions</h3>

<p>HLint enables most Haskell extensions, disabling only those which steal too much syntax (currently Arrows, TransformListComp, XmlSyntax and RegularPatterns). Individual extensions can be enabled or disabled with, for instance, <code>-XArrows</code>, or <code>-XNoMagicHash</code>. The flag <code>-XHaskell98</code> selects Haskell 98 compatibility.</p>

<h3><a id="user-content-emacs-integration" class="anchor" href="#emacs-integration" aria-hidden="true"><span class="octicon octicon-link"></span></a>Emacs Integration</h3>

<p>Emacs integration has been provided by <a href="http://xtalk.msk.su/%7Eott/">Alex Ott</a>. The integration is similar to compilation-mode, allowing navigation between errors. The script is at <a href="https://github.com/ndmitchell/hlint/blob/master/data/hs-lint.el">hs-lint.el</a>, and a copy is installed locally in the data directory. To use, add the following code to the Emacs init file:</p>

<pre><code>(require 'hs-lint)
(defun my-haskell-mode-hook ()
   (local-set-key "\C-cl" 'hs-lint))
(add-hook 'haskell-mode-hook 'my-haskell-mode-hook)
</code></pre>

<h3><a id="user-content-ghci-integration" class="anchor" href="#ghci-integration" aria-hidden="true"><span class="octicon octicon-link"></span></a>GHCi Integration</h3>

<p>GHCi integration has been provided by Gwern Branwen. The integration allows running <code>:hlint</code> from the GHCi prompt. The script is at <a href="http://community.haskell.org/%7Endm/darcs/hlint/data/hlint.ghci">hlint.ghci</a>, and a copy is installed locally in the data directory. To use, add the contents to your <a href="http://www.haskell.org/ghc/docs/latest/html/users_guide/ghci-dot-files.html">GHCi startup file</a>.</p>

<h3><a id="user-content-parallel-operation" class="anchor" href="#parallel-operation" aria-hidden="true"><span class="octicon octicon-link"></span></a>Parallel Operation</h3>

<p>To run HLint on n processors append the flags <code>+RTS -Nn</code>, as described in the <a href="http://www.haskell.org/ghc/docs/latest/html/users_guide/runtime-control.html">GHC user manual</a>. HLint will usually perform fastest if n is equal to the number of physical processors.</p>

<p>If your version of GHC does not support the GHC threaded runtime then install with the command: <code>cabal install --flags="-threaded"</code></p>

<h3><a id="user-content-c-preprocessor-support" class="anchor" href="#c-preprocessor-support" aria-hidden="true"><span class="octicon octicon-link"></span></a>C preprocessor support</h3>

<p>HLint runs the <a href="http://hackage.haskell.org/package/cpphs">cpphs C preprocessor</a> over all input files, by default using the current directory as the include path with no defined macros. These settings can be modified using the flags <code>--cpp-include</code> and <code>--cpp-define</code>. To disable the C preprocessor use the flag <code>-XNoCPP</code>. There are a number of limitations to the C preprocessor support:</p>

<ul>
<li>HLint will only check one branch of an <code>#if</code>, based on which macros have been defined.</li>
<li>Any missing <code>#include</code> files will produce a warning on the console, but no information in the reports.</li>
</ul>

<h3><a id="user-content-unicode-support" class="anchor" href="#unicode-support" aria-hidden="true"><span class="octicon octicon-link"></span></a>Unicode support</h3>

<p>By default, HLint uses the current locale encoding. The encoding can be overridden with either <code>--utf8</code> or <code>--encoding=value</code>. For descriptions of some valid <a href="http://haskell.org/ghc/docs/latest/html/libraries/base/System-IO.html#v%3AmkTextEncoding">encodings see the mkTextEncoding documentation</a>.</p>

<h2><a id="user-content-faq" class="anchor" href="#faq" aria-hidden="true"><span class="octicon octicon-link"></span></a>FAQ</h2>

<h3><a id="user-content-why-are-suggestions-not-applied-recursively" class="anchor" href="#why-are-suggestions-not-applied-recursively" aria-hidden="true"><span class="octicon octicon-link"></span></a>Why are suggestions not applied recursively?</h3>

<p>Consider:</p>

<pre><code>foo xs = concat (map op xs)
</code></pre>

<p>This will suggest eta reduction to <code>concat . map op</code>, and then after making that change and running HLint again, will suggest use of <code>concatMap</code>. Many people wonder why HLint doesn't directly suggest <code>concatMap op</code>. There are a number of reasons:</p>

<ul>
<li>HLint aims to both improve code, and to teach the author better style. Doing modifications individually helps this process.</li>
<li>Sometimes the steps are reasonably complex, by automatically composing them the user may become confused.</li>
<li>Sometimes HLint gets transformations wrong. If suggestions are applied recursively, one error will cascade.</li>
<li>Some people only make use of some of the suggestions. In the above example using concatMap is a good idea, but sometimes eta reduction isn't. By suggesting them separately, people can pick and choose.</li>
<li>Sometimes a transformed expression will be large, and a further hint will apply to some small part of the result, which appears confusing.</li>
<li>Consider <code>f $ (a b)</code>. There are two valid hints, either remove the $ or remove the brackets, but only one can be applied.</li>
</ul>

<h3><a id="user-content-why-arent-the-suggestions-automatically-applied" class="anchor" href="#why-arent-the-suggestions-automatically-applied" aria-hidden="true"><span class="octicon octicon-link"></span></a>Why aren't the suggestions automatically applied?</h3>

<p>If you want to automatically apply suggestions, the <a href="https://rawgithub.com/ndmitchell/hlint/master/hlint.htm#emacs">Emacs integration</a> offers such a feature. However, there are a number of reasons that HLint itself doesn't have an option to automatically apply suggestions:</p>

<ul>
<li>The underlying Haskell parser library makes it hard to modify the code, then print it similarly to the original.</li>
<li>Sometimes multiple transformations may apply.</li>
<li>After applying one transformation, others that were otherwise suggested may become inappropriate.</li>
</ul>

<p>I am intending to develop such a feature, but the above reasons mean it is likely to take some time.</p>

<h3><a id="user-content-why-doesnt-the-compiler-automatically-apply-the-optimisations" class="anchor" href="#why-doesnt-the-compiler-automatically-apply-the-optimisations" aria-hidden="true"><span class="octicon octicon-link"></span></a>Why doesn't the compiler automatically apply the optimisations?</h3>

<p>HLint doesn't suggest optimisations, it suggests code improvements - the intention is to make the code simpler, rather than making the code perform faster. The <a href="http://haskell.org/ghc/">GHC compiler</a> automatically applies many of the rules suggested by HLint, so HLint suggestions will rarely improve performance.</p>

<h3><a id="user-content-why-doesnt-hlint-know-the-fixity-for-my-custom--operator" class="anchor" href="#why-doesnt-hlint-know-the-fixity-for-my-custom--operator" aria-hidden="true"><span class="octicon octicon-link"></span></a>Why doesn't HLint know the fixity for my custom !@%$ operator?</h3>

<p>HLint knows the fixities for all the operators in the base library, but no others. HLint works on a single file at a time, and does not resolve imports, so cannot see fixity declarations from imported modules. You can tell HLint about fixities by putting them in a hint file, or passing them on the command line. For example, pass <code>--with=infixr 5 !@%$</code>, or put all the fixity declarations in a file and pass <code>--hint=fixities.hs</code>. You can also use <a href="https://rawgithub.com/ndmitchell/hlint/master/hlint.htm#find">--find</a> to automatically produce a list of fixity declarations in a file.</p>

<h3><a id="user-content-how-can-i-use---with-or---hint-with-the-default-hints" class="anchor" href="#how-can-i-use---with-or---hint-with-the-default-hints" aria-hidden="true"><span class="octicon octicon-link"></span></a>How can I use <code>--with</code> or <code>--hint</code> with the default hints?</h3>

<p>HLint does not use the default set of hints if custom hints are specified on the command line using <code>--with</code> or <code>--hint</code>. To include the default hints either pass <code>--hint=HLint</code> on the command line, or add import <code>"hint" HLint.HLint</code> in one of the hint files you specify with <code>--hint</code>.</p>

<h3><a id="user-content-why-do-i-sometimes-get-a-note-with-my-hint" class="anchor" href="#why-do-i-sometimes-get-a-note-with-my-hint" aria-hidden="true"><span class="octicon octicon-link"></span></a>Why do I sometimes get a "Note" with my hint?</h3>

<p>Most hints are perfect substitutions, and these are displayed without any notes. However, some hints change the semantics of your program - typically in irrelevant ways - but HLint shows a warning note. HLint does not warn when assuming typeclass laws (such as <code>==</code> being symmetric). Some notes you may see include:</p>

<ul>
<li><strong>Increases laziness</strong> - for example <code>foldl (&amp;&amp;) True</code> suggests <code>and</code> including this note. The new code will work on infinite lists, while the old code would not. Increasing laziness is usually a good idea.</li>
<li><strong>Decreases laziness</strong> - for example <code>(fst a, snd a)</code> suggests a including this note. On evaluation the new code will raise an error if a is an error, while the old code would produce a pair containing two error values. Only a small number of hints decrease laziness, and anyone relying on the laziness of the original code would be advised to include a comment.</li>
<li><strong>Removes error</strong> - for example foldr1 (&amp;&amp;) suggests and including the note "Removes error on []". The new code will produce <code>True</code> on the empty list, while the old code would raise an error. Unless you are relying on the exception thrown by the empty list, this hint is safe - and if you do rely on the exception, you would be advised to add a comment. </li>
</ul>

<h3><a id="user-content-what-is-the-difference-between-error-and-warning" class="anchor" href="#what-is-the-difference-between-error-and-warning" aria-hidden="true"><span class="octicon octicon-link"></span></a>What is the difference between error and warning?</h3>

<p>Every hint has a severity level:</p>

<ul>
<li><strong>Error</strong> - for example <code>concat (map f x)</code> suggests <code>concatMap f x</code> as an "error" severity hint. From a style point of view, you should always replace a combination of <code>concat</code> and <code>map</code> with <code>concatMap</code>. Note that both expressions are equivalent - HLint is reporting an error in style, not an actual error in the code.</li>
<li><strong>Warning</strong> - for example <code>x !! 0</code> suggests head x as a "warning" severity hint. Typically head is a simpler way of expressing the first element of a list, especially if you are treating the list inductively. However, in the expression <code>f (x !! 4) (x !! 0) (x !! 7)</code>, replacing the middle argument with <code>head</code> makes it harder to follow the pattern, and is probably a bad idea. Warning hints are often worthwhile, but should not be applied blindly.</li>
</ul>

<p>The difference between error and warning is one of personal taste, typically my personal taste. If you already have a well developed sense of Haskell style, you should ignore the difference. If you are a beginner Haskell programmer you may wish to focus on error hints before warning hints.</p>

<h2><a id="user-content-customizing-the-hints" class="anchor" href="#customizing-the-hints" aria-hidden="true"><span class="octicon octicon-link"></span></a>Customizing the hints</h2>

<p>Many of the hints that are applied by HLint are contained in Haskell source files which are installed in the data directory by Cabal. These files may be edited, to add library specific knowledge, to include hints that may have been missed, or to ignore unwanted hints.</p>

<h3><a id="user-content-choosing-a-package-of-hints" class="anchor" href="#choosing-a-package-of-hints" aria-hidden="true"><span class="octicon octicon-link"></span></a>Choosing a package of hints</h3>

<p>By default, HLint will use the <code>HLint.hs</code> file either from the current working directory, or from the data directory. Alternatively, hint files can be specified with the <code>--hint</code> flag. HLint comes with a number of hint packages:</p>

<ul>
<li><strong>Default</strong> - these are the hints that are used by default, covering most of the base libraries.</li>
<li><strong>Dollar</strong> - suggests the replacement <code>a $ b $ c</code> with <code>a . b $ c</code>. This hint is especially popular on the <a href="http://www.haskell.org/haskellwiki/IRC_channel">#haskell IRC channel</a>.</li>
<li><strong>Generalise</strong> - suggests replacing specific variants of functions (i.e. <code>map</code>) with more generic functions (i.e. <code>fmap</code>).</li>
</ul>

<p>As an example, to check the file <code>Example.hs</code> with both the default hints and the dollar hint, I could type: <code>hlint Example.hs --hint=Default --hint=Dollar</code>. Alternatively, I could create the file <code>HLint.hs</code> in the working directory and give it the contents:</p>

<pre><code>import "hint" HLint.Default
import "hint" HLint.Dollar
</code></pre>

<h3><a id="user-content-ignoring-hints" class="anchor" href="#ignoring-hints" aria-hidden="true"><span class="octicon octicon-link"></span></a>Ignoring hints</h3>

<p>Some of the hints are subjective, and some users believe they should be ignored. Some hints are applicable usually, but occasionally don't always make sense. The ignoring mechanism provides features for suppressing certain hints. Ignore directives can either be written as pragmas in the file being analysed, or in the hint files. Examples of pragmas are:</p>

<ul>
<li><code>{-# ANN module "HLint: ignore Eta reduce" #-}</code> - ignore all eta reduction suggestions in this module (use <code>module</code> literally, not the name of the module).</li>
<li><code>{-# ANN myFunction "HLint: ignore" #-}</code> - don't give any hints in the function <code>myFunction</code>.</li>
<li><code>{-# ANN myFunction "HLint: error" #-}</code> - any hint in the function <code>myFunction</code> is an error.</li>
<li><code>{-# ANN module "HLint: error Use concatMap" #-}</code> - the hint to use concatMap is an error.</li>
<li><code>{-# ANN module "HLint: warn Use concatMap" #-}</code> - the hint to use concatMap is a warning.</li>
</ul>

<p>Ignore directives can also be written in the hint files:</p>

<ul>
<li><code>ignore "Eta reduce"</code> - suppress all eta reduction suggestions.</li>
<li><code>ignore "Eta reduce" = MyModule1 MyModule2</code> - suppress eta reduction hints in the <code>MyModule1</code> and <code>MyModule2</code> modules.</li>
<li><code>ignore = MyModule.myFunction</code> - don't give any hints in the function <code>MyModule.myFunction</code>.</li>
<li><code>error = MyModule.myFunction</code> - any hint in the function <code>MyModule.myFunction</code> is an error.</li>
<li><code>error "Use concatMap"</code> - the hint to use <code>concatMap</code> is an error.</li>
<li><code>warn "Use concatMap"</code> - the hint to use <code>concatMap</code> is a warning.</li>
</ul>

<p>These directives are applied in the order they are given, with later hints overriding earlier ones.</p>

<h3><a id="user-content-adding-hints" class="anchor" href="#adding-hints" aria-hidden="true"><span class="octicon octicon-link"></span></a>Adding hints</h3>

<p>The hint suggesting <code>concatMap</code> is defined as:</p>

<pre><code>error = concat (map f x) ==&gt; concatMap f x
</code></pre>

<p>The line can be read as replace <code>concat (map f x)</code> with <code>concatMap f x</code>. All single-letter variables are treated as substitution parameters. For examples of more complex hints see the supplied hints file. In general, hints should not be given in point free style, as this reduces the power of the matching. Hints may start with <code>error</code> or <code>warn</code> to denote how severe they are by default. If you come up with interesting hints, please submit them for inclusion.</p>

<p>You can search for possible hints to add from a source file with the <code>--find</code> flag, for example:</p>

<pre><code>$ hlint --find=src/Utils.hs
-- hints found in src/Util.hs
warn = null (intersect a b) ==&gt; disjoint a b
warn = dropWhile isSpace ==&gt; trimStart
infixr 5 !:
</code></pre>

<p>These hints are suitable for inclusion in a custom hint file. You can also include Haskell fixity declarations in a hint file, and these will also be extracted. If you pass only <code>--find</code> flags then the hints will be written out, if you also pass files/folders to check, then the found hints will be automatically used when checking.</p>
